### InferenceJobDB

ORM model in `backend/services/db_model.py`. Table: `inference_jobs`.

| Column | Type | Constraints |
|---|---|---|
| `job_id` | String(36) | PK (UUID as string) |
| `status` | String(20) | NOT NULL, default `"pending"` |
| `result` | JSON | nullable |
| `error` | Text | nullable |
| `created_at` | TIMESTAMPTZ | NOT NULL |
| `completed_at` | TIMESTAMPTZ | nullable |

**Status state machine**:
```
pending → processing → completed
                    ↘ failed
```

**Migration**: `add_inference_jobs` migration adds this table. The name implies it was added after the initial schema — both tables are in the ORM today but autogenerate of two sequential migrations produces this ordering.

## New Entities (Runtime Only — No Additional Tables)

### Rate Limit Window (in-memory / Redis)

Not a database table. Managed by `slowapi` using Redis as the storage backend when configured, or in-process memory for single-worker deployments.

| Attribute | Type | Description |
|---|---|---|
| `key` | string | `"{remote_ip}:{endpoint}"` |
| `count` | integer | Request count within window |
| `window_start` | timestamp | Rolling window start |
| `limit` | integer | Maximum requests per window (10) |
| `window_seconds` | integer | Window duration (60) |

### Request Trace (transient — per HTTP request)

Not persisted. Lives in `request.state` for the duration of a single HTTP request lifecycle.

| Attribute | Type | Source |
|---|---|---|
| `request_id` | UUID string | From `X-Request-ID` header, or generated |

Exposed via:
- `request.state.request_id` (server-side)
- `X-Request-ID` response header (client-visible)

### Celery Task (broker + result backend: Redis)

Not a PostgreSQL table. Managed by Celery's Redis-backed result store.

| Attribute | Type | Description |
|---|---|---|
| `task_id` | string | Celery's internal UUID (not the same as `job_id`) |
| `job_id` | string | Our `inference_jobs.job_id` passed as argument |
| `state` | string | Celery states: PENDING → STARTED → SUCCESS/FAILURE |
| `result` | JSON | Celery result (we persist separately to PostgreSQL) |

**Note**: The Celery task state and the `InferenceJobDB.status` are maintained independently. The PostgreSQL job table is the source of truth for API consumers; Celery state is internal only.
